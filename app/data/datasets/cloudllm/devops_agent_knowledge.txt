SECTION 1: ACTION TYPES
================================================================================

The DevOps Agent supports 7 action types. Each action is output as a JSON block.

ACTION 1: execute (script or terraform)
- Runs a bash script on a remote VM via SCP+SSH
- Runs terraform locally on the server with cloud credentials from vault
- Fields: action, type (script|terraform), path, args (optional), explanation
- Example: {"action":"execute","type":"script","path":"redis/install_redis.sh","explanation":"Installing Redis"}
- Example with args: {"action":"execute","type":"script","path":"network/open_port_scanner.sh","args":"192.168.1.1 1 1024 50","explanation":"Scanning ports"}
- Example terraform: {"action":"execute","type":"terraform","path":"aws/terraform_aws_s3_bucket","explanation":"Creating S3 bucket"}

ACTION 2: git_clone
- Clones a git repository into a session directory
- Fields: action, repo_url, branch, explanation
- Example: {"action":"git_clone","repo_url":"https://github.com/user/repo.git","branch":"main","explanation":"Cloning repository"}

ACTION 3: write_file
- Creates or edits a file in the session directory
- Fields: action, file_path, content, explanation
- Example: {"action":"write_file","file_path":"Dockerfile","content":"FROM node:20\nWORKDIR /app\nCOPY . .\nCMD [\"node\",\"app.js\"]","explanation":"Creating Dockerfile"}

ACTION 4: git_push
- Commits and pushes changes to a git branch
- Fields: action, branch, message, explanation
- Example: {"action":"git_push","branch":"feat/add-docker","message":"Add Dockerfile","explanation":"Pushing changes"}

ACTION 5: git_pr
- Creates a pull request on GitHub/GitLab
- Fields: action, head_branch, base_branch, title, body, explanation
- Example: {"action":"git_pr","head_branch":"feat/add-docker","base_branch":"main","title":"Add Docker support","body":"Adds Dockerfile and docker-compose","explanation":"Creating PR"}

ACTION 6: execute_cli (vxcli command)
- ONLY triggered when user explicitly mentions "vxcli" or "vx" in their prompt
- Generates a vxcli CLI command for the user to execute
- Fields: action, command, explanation
- Example: {"action":"execute_cli","command":"vxcli deploy --app myapp --cloud aws --service ec2","explanation":"Deploy to AWS EC2"}
- Example: {"action":"execute_cli","command":"vxcli auth login -k xc_dev_mykey -u myuser","explanation":"Login to platform"}
- IMPORTANT: Do NOT generate execute_cli unless the prompt contains "vxcli" or "vx"

ACTION 7: multi_step (chained workflow)
- Triggered when user describes a multi-step workflow
- Contains a steps array with 2-5 sequential actions
- Fields: action, steps (array), explanation
- Example: {"action":"multi_step","steps":[{"action":"git_clone","repo_url":"...","branch":"main"},{"action":"execute","type":"script","path":"deployments/deploy_dockerized_fastapi.sh"}],"explanation":"Clone and deploy"}

SECTION 5: EXECUTION ROUTING
================================================================================

RULE 1: All terraform modules run LOCALLY on the Go server (terraform talks to cloud APIs).
RULE 2: All bash scripts run on the REMOTE VM via SCP + SSH.
RULE 3: Git operations (clone, push, PR) run LOCALLY on the Go server.
RULE 4: File write operations happen LOCALLY in the session directory.
RULE 5: Cloud credentials are loaded from vault automatically.
RULE 6: SSH credentials are loaded from vault using the key_pair_name.

SECTION 6: MULTI-ACTION WORKFLOWS
================================================================================

Users can request multi-step workflows. Output max 5 actions in sequence:

Example: "Clone my repo, install Nginx, deploy the site"
1. {"action":"git_clone","repo_url":"...","branch":"main","explanation":"Cloning repo"}
2. {"action":"execute","type":"script","path":"utils/install_nginx.sh","explanation":"Installing Nginx"}
3. {"action":"execute","type":"script","path":"vm/deploy_frontend.sh","explanation":"Deploying site"}

Example: "Set up monitoring with Grafana and Prometheus"
1. {"action":"execute","type":"terraform","path":"docker/terraform_docker_linux_grafana","explanation":"Deploying Grafana"}
2. {"action":"execute","type":"terraform","path":"docker/terraform_docker_linux_prometheus","explanation":"Deploying Prometheus"}

Example: "Create a Dockerfile, push to GitHub, create PR"
1. {"action":"write_file","file_path":"Dockerfile","content":"...","explanation":"Creating Dockerfile"}
2. {"action":"git_push","branch":"feat/add-docker","message":"Add Dockerfile","explanation":"Pushing"}
3. {"action":"git_pr","head_branch":"feat/add-docker","base_branch":"main","title":"Add Docker","body":"...","explanation":"Creating PR"}

SECTION 7: DESTRUCTIVE ACTIONS (require confirmation)
================================================================================

These actions are flagged as destructive and require user confirmation:
- reboot, shutdown (server)
- remove_file, kill_port
- Any script with "destroy", "delete", "remove", "uninstall" in the path
- terraform destroy operations

SECTION 8: INTENT CLASSIFICATION
================================================================================

The SLM should classify user prompts into these intents:
- execute_script: User wants to run a bash script
- execute_terraform: User wants to provision infrastructure via terraform
- git_clone: User wants to clone a repository
- git_push: User wants to commit and push code
- git_pr: User wants to create a pull request
- write_file: User wants to create or edit a file
- system_check: User wants to scan/check their server (ports, disk, RAM)
- deploy_service: User wants to deploy a service (FastAPI, React, Next.js, Django, static site)
- deploy_container: User wants to deploy a Docker container
- deploy_kubernetes: User wants to deploy to a Kubernetes cluster
- deploy_agent: User wants to deploy an AI agent
- setup_cicd: User wants to set up CI/CD pipeline (GitHub Actions, Jenkins, GitLab CI, Gitea, Forgejo)
- run_ansible: User wants to run an Ansible playbook
- pipeline_operation: User wants to manage CI/CD pipelines (create, trigger, pause, resume, view logs)
- setup_webhook: User wants to configure git webhooks
- multi_action_workflow: User describes a complex multi-step deployment workflow
- execute_cli: User explicitly mentions "vxcli" or "vx" — generate a CLI command
- generate_workflow: User wants to generate a CI/CD workflow (GitHub Actions, GitLab CI)
- other: General question, not actionable

CRITICAL ROUTING RULE for execute_cli:
- ONLY classify as execute_cli when the prompt contains "vxcli" or "vx"
- "deploy my app to AWS" → execute_script or deploy_service (NOT execute_cli)
- "use vx to deploy my app to AWS" → execute_cli
- "vxcli deploy --app myapp" → execute_cli
- This prevents vxcli from cannibalizing script/terraform/deployment intents

Confidence scoring:
- 0.9+ : Clear, unambiguous request with specific tool/script mentioned
- 0.7-0.9 : Clear intent but tool needs to be inferred
- 0.5-0.7 : Ambiguous, may need clarification
- <0.5 : Not a DevOps request
